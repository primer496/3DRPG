# FinalRPG 项目代码结构说明

本文档用**通俗语言**说明整个项目的代码结构，并对 **async/await、Task、轮询** 等概念做简要解释，方便不常接触异步 API 的读者理解。

---

## 一、项目概览

这是一个 **Unity 2D/3D RPG 示例项目**，当前代码主要集中在：

- **玩家控制**：键盘 A/D 或左右方向键移动，空格跳跃
- **分层状态机 (HSM)**：用“状态树”管理玩家行为（站立、行走、空中等）
- **状态切换时的“活动”**：进入/离开某个状态时可以执行异步任务（例如改颜色、延迟等），**不阻塞游戏主循环**

没有独立的战斗、存档、复杂 UI 等系统，所有逻辑都在 **Assets/Scripts/Unity-HSM** 下。

## 三、游戏主循环（一帧里发生了什么）

Unity 每帧会调用 `Update()`，本项目的主循环可以概括为：

```
┌─────────────────────────────────────────────────────────────────┐
│  PlayerStateDriver.Update() 每帧执行                             │
├─────────────────────────────────────────────────────────────────┤
│  1. 读输入：A/D、空格 → ctx.move.x, ctx.jumpPressed              │
│  2. 检测是否着地：Physics.CheckSphere → ctx.grounded             │
│  3. 驱动状态机：machine.Tick(Time.deltaTime)  ← 核心            │
│  4. （可选）状态变化时打日志                                      │
└─────────────────────────────────────────────────────────────────┘
```

`machine.Tick()` 内部：

- **若当前没有在“切换状态”**：  
  `TransitionSequencer.Tick()` 会调用 `StateMachine.InternalTick(deltaTime)`，从根状态一路 `Update` 到当前**叶子状态**（Idle / Move / Airborne）。  
  在 `State.Update()` 里会调用 `GetTransition()`，若返回非空就**请求一次状态切换**。
- **若正在切换状态**：  
  不执行正常的状态 `Update`，只执行 `TransitionSequencer` 的“当前阶段”（先退出阶段，再切状态，再进入阶段）。  
  每个阶段里可能有多步 **Task**（例如多个 Activity 的 Activate/Deactivate），这些 Task 是**异步的**，但主循环**不 await**，而是下一帧再调用 `sequencer.Update()`，看当前阶段是否**全部完成**（轮询）。

所以：  
- **主循环是同步、按帧执行的**，没有在 `Update` 里写 `await`。  
- **异步**只发生在“切换阶段的步骤”里（例如 `ColorPhaseActivity`、`DelayActivationActivity`），用 **Task** 表示“将来会完成的工作”；  
  是否完成由 **TransitionSequencer** 每帧用 `Update()` **轮询**检查。

## 四、分层状态机 (HSM) 结构

### 4.1 概念简述

- **状态** 组织成**树**：有父状态、子状态；同一时刻只有一条从根到**叶子**的路径是“当前状态”。
- **叶子状态** 才是具体行为：例如 Idle、Move、Airborne。
- **切换**：由当前状态的 `GetTransition()` 返回“要去的状态”；  
  实际切换时，会算两个状态的**最近公共祖先 (LCA)**，先沿当前路径 **Exit** 到 LCA，再沿目标路径从 LCA **Enter** 到目标。

### 4.2 本项目的状态树

```
                    PlayerRoot
                    /        \
              Grounded      Airborne
              /      \
           Idle      Move
```

- **PlayerRoot**：根；根据 `ctx.grounded` 在 Grounded / Airborne 之间选。
- **Grounded**：地面；默认进 Idle；根据 `ctx.move.x` 在 Idle ↔ Move；根据 `ctx.jumpPressed` 或离开地面 → Airborne。
- **Idle**：站立；`OnEnter` 把水平速度置 0；有左右输入 → Move。
- **Move**：移动；`OnUpdate` 里用 `ctx.move.x * moveSpeed` 和加速度更新 `ctx.velocity.x`；无输入或离地 → Idle 或 Airborne。
- **Airborne**：空中；落地后 `ctx.grounded == true` → Grounded。

**Grounded** 和 **Airborne** 各挂了一个 **ColorPhaseActivity**（改材质颜色），用于在进入/离开该状态时做“活动”，便于观察状态切换。

### 4.3 状态与“活动”(Activity)

每个 **State** 可以挂多个 **IActivity**。  
当状态**被进入**时，会按“进入链”依次 **Activate** 这些活动；**被退出**时，会按“退出链”依次 **Deactivate**。  
Activate/Deactivate 的签名是：

- `Task ActivateAsync(CancellationToken ct)`
- `Task DeactivateAsync(CancellationToken ct)`

也就是说，**活动**的开启/关闭可以是**异步的**（例如等待几秒、播一段动画）。  
但在本项目中，主循环并不直接 `await` 这些 Task，而是由 **TransitionSequencer** 把多个步骤放进 **ParallelPhase** 或 **SequentialPhase**，每帧调用 `Update()`，用 **Task 是否都完成** 来推进阶段。这就是下面要说的“轮询式异步”。

## 五、以 TransitionSequencer 为核心的过渡逻辑

这一节专门讲**状态切换**时发生了什么，以及 **TransitionSequencer** 如何把“退出活动 → 改状态树 → 进入活动”拆成可异步的三段，并用**轮询**在每帧里推进。

### 5.1 谁在什么时候发起“切换”？

- 每帧 **StateMachine.Tick** 会调用 **TransitionSequencer.Tick**。
- 若**当前没有在过渡中**（`sequencer == null`），则执行 **InternalTick(deltaTime)**，即从根状态一路 **Update** 到当前叶子。
- 在某个状态的 **State.Update** 里会调 **GetTransition()**；若返回**非 null**（表示“这一帧要切到另一个状态”），就立刻调用：
  - **Machine.Sequencer.RequestTransition(this, t)**  
  这里 `this` 是**发现需要切换的那个状态**（往往是当前叶子，例如 Idle），`t` 是目标状态（例如 Airborne）。
- 所以：**“逻辑上决定要切换”的时机** = 某状态 `GetTransition()` 返回非 null 的那一帧；**发起者** = TransitionSequencer 的 **RequestTransition(State from, State to)**。

### 5.2 RequestTransition 做了什么？

```text
RequestTransition(State from, State to)
├── 若 to==null 或 from==to → 直接 return
├── 若当前已经在过渡中（sequencer != null）→ 只把 (from, to) 存进 pending，等本次过渡结束后再处理
└── 否则 → BeginTransition(from, to)
```

- **正在过渡时再请求**：不立刻再开一次过渡，只记一个 **pending**，当前过渡在 **EndTransition** 里会检查 pending 并再调 **BeginTransition**，保证同一时间只跑“一段”过渡，避免交错。

### 5.3 BeginTransition：LCA、退出链、进入链

**目的**：从“当前状态”到“目标状态”的切换，只**退出**需要退出的状态、只**进入**需要进入的状态，公共祖先不动。

- **LCA = Lowest Common Ancestor**：`from` 和 `to` 在状态树上的**最近公共祖先**。  
  例：from = Idle，to = Airborne → 路径 Idle→Grounded→PlayerRoot，Airborne→PlayerRoot → **LCA = PlayerRoot**。
- **退出链 exitChain**：从 `from` 一路沿 **Parent** 走到 **LCA（不包含 LCA）**，顺序是**自下而上**（先叶子，再父）。  
  例：from=Idle, LCA=PlayerRoot → exitChain = [Idle, Grounded]。  
  这些状态上的 **Activity** 要在切换前 **Deactivate**（先关掉“旧分支”上的活动）。
- **进入链 enterChain**：从 `to` 一路沿 Parent 走到 LCA（不包含 LCA），顺序是**自上而下**（先靠近 LCA 的，再叶子）。  
  例：to=Airborne, LCA=PlayerRoot → enterChain = [Airborne]。  
  这些状态上的 **Activity** 要在切换后 **Activate**（再打开“新分支”上的活动）。

所以：**过渡分三阶段**——  
1) 对 exitChain 里所有“当前 Active 的活动”做 **Deactivate**（异步，用 Phase 跑）；  
2) **ChangeState(from, to)**：真正改状态树（Exit 旧分支、Enter 新分支）；  
3) 对 enterChain 里所有“当前 Inactive 的活动”做 **Activate**（异步，再一个 Phase）。

### 5.4 三阶段在代码里怎么串起来？

- **阶段 1（退出活动）**  
  - **GatherPhaseSteps(exitChain, deactivate: true)**：收集所有“Mode == Active”的 Activity 的 **DeactivateAsync**，做成一批 **PhaseStep**。  
  - 用 **ParallelPhase**（或 SequentialPhase）包起来，**Start()** 时把这些 Task 都启动。  
  - **sequencer** 指向这个 Phase；**nextPhase** 赋值为一个 lambda，里面做：调用 **Machine.ChangeState(from, to)**，再对 enterChain 做 **GatherPhaseSteps(enterChain, deactivate: false)**，再建一个 Phase 跑 **Activate**，并 **Start()**，同时把 **nextPhase 置空**（表示下一阶段完成后就结束过渡）。

- **阶段 2 + 3**  
  - 阶段 1 的 Phase 在 **Tick()** 里被轮询；当 **sequencer.Update()** 返回 true（所有 Deactivate 的 Task 都完成）时，执行 **nextPhase()**。  
  - 在 nextPhase 里：先 **ChangeState**（改状态树），再建“进入活动”的 Phase，**Start()**，此时 **sequencer** 指向这个新 Phase，**nextPhase** 被置为 null。  
  - 之后每帧 **Tick()** 继续轮询这个新 **sequencer**；当 **sequencer.Update()** 再返回 true（所有 Activate 完成）时，因为 **nextPhase == null**，就调 **EndTransition()**：清空 sequencer，若有 pending 再 **BeginTransition(pending)**。

所以：**没有“从 State 到 PlayerRoot 的转换”**——LCA 只是用来算“从哪到哪的链”；**状态树结构的改变**只在 **ChangeState** 里通过 **Exit/Enter** 完成。

### 5.5 Tick() 在过渡期间的行为（一图看懂）

```text
每帧：TransitionSequencer.Tick(deltaTime)
│
├─ 若 sequencer == null（没有在过渡）
│      → Machine.InternalTick(deltaTime)   // 正常状态 Update，可能触发 RequestTransition
│      → return
│
└─ 若 sequencer != null（正在过渡）
       → 调用 sequencer.Update()
       │     → ParallelPhase/SequentialPhase 里检查所有 Task 是否都完成
       │     → 未完成：return false
       │     → 都完成：return true
       │
       → 若 Update() 为 true：
       │     → 若 nextPhase != null：执行 nextPhase()（ChangeState + 启动“进入活动”Phase，nextPhase 置空）
       │     → 若 nextPhase == null：EndTransition()（清空 sequencer，若有 pending 再 BeginTransition）
       → 本帧不执行 InternalTick（过渡期间不跑正常状态逻辑）
```

一句话：**过渡中**只做“轮询当前 Phase 是否完成 → 完成则进入下一阶段或结束”；**非过渡**才跑状态机的 Update。

### 5.6 小结（TransitionSequencer 在干什么）

| 概念 | 作用 |
|------|------|
| **RequestTransition(from, to)** | 发起一次从 from 到 to 的切换；若已在过渡则只记 pending。 |
| **LCA** | 算“退出链”和“进入链”的边界，只对两条链上的状态做 Exit/Enter 和 Activity 的 Deactivate/Activate。 |
| **exitChain / enterChain** | 退出链上的活动先 Deactivate（Phase 1）；进入链上的活动在 ChangeState 之后 Activate（Phase 3）。 |
| **ChangeState** | 在 Phase 1 完成后被 nextPhase 调用：Exit 旧分支到 LCA，再 Enter 新分支到 to。 |
| **sequencer / nextPhase** | sequencer = 当前正在跑的“阶段”（一批 Task）；完成后若 nextPhase 非空就执行（进入 Phase 2+3），否则 EndTransition。 |
| **Tick()** | 有 sequencer 就只轮询当前阶段、不跑状态 Update；没有就正常跑 InternalTick。 |

这样，**状态切换**和**活动开关**都经过 TransitionSequencer，既保证顺序（先关旧活动 → 改状态树 → 再开新活动），又支持活动里用异步（Task），主循环用轮询不阻塞。

## 六、异步 (async/Task) 在本项目中的用法（通俗版）

### 6.1 什么是 async / Task（简要）

- **async**：标记一个方法里“会有异步操作”，方法可以 **await** 别处返回的 **Task**。  
  `await` 时，当前方法会“让出”，等 Task 完成后再继续，**不会阻塞调用线程**。
- **Task**：表示“一件会在将来完成的事”（可能已经完成）。  
  例如 `Task.Delay(1000)` 表示“大约 1 秒后完成”的 Task。

在普通 C# 程序里，我们可能这样写：

```csharp
await Task.Delay(1000);  // 等 1 秒再往下执行
```

但在 **Unity 主循环**里，通常**不能**在 `Update()` 里直接 `await`，因为：

- `Update()` 需要每帧快速返回，Unity 靠每帧调用它来驱动游戏；
- 若在 `Update()` 里 `await`，这一帧就会“卡”在那里，直到 Task 完成，帧率会崩。

所以本项目采用另一种方式：**在子逻辑里启动 Task，主循环不 await，而是每帧问“做完了吗？”**。

### 6.2 本项目怎么做：轮询 (Polling)

- **谁在跑 Task？**  
  **TransitionSequencer** 在开始“退出阶段”或“进入阶段”时，会创建 **ParallelPhase** 或 **SequentialPhase**，并调用它们的 `Start()`。  
  `Start()` 里会**调用**每个步骤（即 `PhaseStep`），每个步骤返回一个 **Task**（例如 `activity.ActivateAsync(ct)`），这些 Task 在后台进行。

- **主循环在干什么？**  
  每帧 `TransitionSequencer.Tick()` 被调用时，若当前处于“切换中”，就只调用当前阶段的 `sequencer.Update()`。  
  `Update()` 里（以 **ParallelPhase** 为例）只是检查：  
  `tasks.TrueForAll(t => t.IsCompleted)`  
  即：**所有 Task 是否都已完成**。  
  - 若没完成，`Update()` 返回 false，下一帧再检查；  
  - 若都完成了，`Update()` 返回 true，TransitionSequencer 进入下一步（换状态或激活进入链，或结束切换）。

所以：

- **主循环**：永远只是“调用 `Tick()` → 里面再调用 `Update()` 看完成没”的**同步、按帧执行**的逻辑，**没有 await**。
- **异步**：只发生在 Activity 的 `ActivateAsync` / `DeactivateAsync` 内部（例如 `DelayActivationActivity` 里 `await Task.Delay(...)`）。  
  这些 Task 由 .NET 线程池/定时器在后台推进，主线程每帧只做一次“完成了吗？”的检查，这就是**轮询**。

可以理解为：  
**“异步” = 事情在后台做；“轮询” = 主循环每帧问一句“做完了没”，做完了再继续下一步。**  
这样既不用在 `Update()` 里写 `await`，又能享受“等待一段时间”“等待多个操作”等异步能力。

### 6.3 相关类小结

| 类/接口 | 作用（通俗说） |
|--------|----------------|
| **IActivity** | “活动”接口：能异步地“开启”和“关闭”。 |
| **Activity** | 基类：默认 Activate/Deactivate 立刻完成（await Task.CompletedTask）。 |
| **ColorPhaseActivity** | 进入状态时把材质设为 enterColor，离开时设为 exitColor（本项目里实际是同步的，但签名仍是 async Task）。 |
| **DelayActivationActivity** | 真正用了 `await Task.Delay(seconds)`，延迟几秒再激活。 |
| **PhaseStep** | 委托：`CancellationToken -> Task`，表示“一步操作”（通常就是调用某个 Activity 的 Activate/Deactivate）。 |
| **ISequence** | 一个“阶段”：可以 Start，然后通过 Update() 轮询是否完成。 |
| **ParallelPhase** | 同时启动多个 PhaseStep 的 Task，Update() 时检查是否**全部**完成。 |
| **SequentialPhase** | 按顺序执行多个 PhaseStep，当前一个的 Task 完成后再启动下一个，Update() 检查当前步骤是否完成。 |

## 七、依赖关系简图（谁用谁）

```
PlayerStateDriver
    ├── 使用 StateMachine, StateMachineBuilder, PlayerRoot, PlayerContext
    ├── 读输入 (InputSystem)、检测地面 (Physics)
    └── 每帧 machine.Tick(deltaTime)

StateMachine
    ├── 持有 State Root, TransitionSequencer
    ├── Tick() → Sequencer.Tick() 或 InternalTick()
    └── ChangeState(from, to) 被 TransitionSequencer 在阶段之间调用

TransitionSequencer
    ├── 持有 StateMachine
    ├── 用 LCA、StatesToExit、StatesToEnter 算退出/进入链
    ├── 用 GatherPhaseSteps 收集 PhaseStep（调用 IActivity 的 Activate/Deactivate）
    ├── 用 ParallelPhase / SequentialPhase 执行这些步骤，每帧 Update() 轮询
    └── 阶段完成后调用 Machine.ChangeState 或进入下一阶段

State
    ├── 持有 StateMachine, Parent, ActiveChild, List<IActivity>
    ├── Enter/Exit/Update 内部会调用 OnEnter/OnExit/OnUpdate、GetInitialState、GetTransition
    └── GetTransition() 若返回非空，会 Machine.Sequencer.RequestTransition(this, target)

PlayerRoot, Grounded, Idle, Move, Airborne
    └── 都是 State 子类，构造时挂需要的 Activity（如 ColorPhaseActivity）
```

外部依赖：**UnityEngine**、**UnityEngine.InputSystem**、**UnityUtils**（如 `GetOrAdd`、`Logwin.Log`，不在本仓库）。

## 九、小结

- **项目类型**：Unity 小示例，主打**分层状态机 + 玩家移动/跳跃**。  
- **代码集中地**：`Assets/Scripts/Unity-HSM`，命名空间 **HSM**。  
- **主循环**：`PlayerStateDriver.Update()` → 读输入、测地面 → `machine.Tick(deltaTime)`；**全程同步、不 await**。  
- **异步**：只用于状态切换时的“活动”步骤（Activate/Deactivate），通过 **Task** 执行；主循环用 **轮询**（每帧 `Update()` 检查 Task 是否完成）推进，不阻塞帧循环。  
- **状态树**：PlayerRoot → Grounded（Idle / Move）与 Airborne；切换时按 LCA 做 Exit/Enter，并配合 TransitionSequencer 做带异步活动的阶段执行。

如果你希望某一块再展开（例如 LCA 怎么算、或者自己加一个带延迟的状态活动），可以指定文件名或类名，我可以按文件/类逐行说明。
